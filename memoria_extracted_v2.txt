Apartado 1: Análisis del Problema y Justificación

1.1 Identificación de la brecha de mercado

La comarca del Vallès Oriental presenta una particularidad demográfica significativa:
una alta concentración de ciudadanos de origen dominicano que, hasta la fecha, no
cuenta con una oferta gastronómica especializada en su sector. Actualmente, la
demanda de productos y platos tradicionales dominicanos —esenciales para la

cohesión cultural y el bienestar de esta comunidad— no está siendo cubierta en la
zona.

1.2 Análisis de la problemática logística

Esta carencia de oferta local genera una ineficiencia en el consumo de los usuarios de
este segmento:

    •   Desplazamiento forzado: El consumidor debe trasladarse hasta el centro de
        Barcelona —una distancia media de 30 kilómetros— para acceder a servicios de
        comida que respondan a sus expectativas culturales y nostálgicas.







    •   Coste de oportunidad: Este desplazamiento implica una inversión de tiempo
        (promedio de 1 hora por trayecto) y un coste económico derivado del transporte,

        lo cual limita la frecuencia de consumo y reduce el acceso a estos servicios a
        ocasiones excepcionales en lugar de ser una opción cotidiana.



    •   Impacto en el sector: La ausencia de servicios de proximidad en el Vallès
        Oriental representa una oportunidad de negocio desaprovechada, dejando un
        mercado cautivo sin una solución logística eficiente.

1.3 Propuesta de Valor: "El Fogón Dominicano"

El Fogón Dominicano surge como respuesta directa a esta necesidad de
descentralización. El proyecto no solo busca ofrecer un producto gastronómico, sino
optimizar el acceso al mismo mediante una solución digital de proximidad:

    •   Optimización de la cadena de suministro: Permite gestionar la producción
        según la demanda real mediante un sistema de reservas, reduciendo el
        desperdicio alimentario y garantizando frescura.

    •   Proximidad operativa: Elimina la barrera de los 30 km de distancia,
        permitiendo que la comunidad dominicana del Vallès Oriental acceda a su
        gastronomía local sin necesidad de grandes desplazamientos.

    •   Digitalización del servicio: Al implementar una plataforma de reservas en
        línea, se moderniza un servicio tradicional, facilitando la gestión de pedidos y

Apartado 2: Arquitectura Técnica y Solución Tecnológica

2.1 Stack Tecnológico

Para garantizar la fiabilidad, escalabilidad y seguridad del servicio, se ha seleccionado
un stack tecnológico estándar en la industria, optimizado para entornos de alta
disponibilidad:

    •   Framework Backend (Laravel): Se eligió por su robustez en la gestión de la
        lógica de negocio y su ecosistema de seguridad integrado. Laravel permite una
        arquitectura modular que facilita el mantenimiento y la evolución del sistema.

        base de datos relacional. Esta elección responde a la necesidad de mantener
        una estructura de datos altamente consistente, fundamental para la gestión de
        inventario y el historial de pedidos. La integridad referencial de MySQL garantiza
        la coherencia entre usuarios, platos y reservas.



    •   Sistema de Vistas (Blade Engine): Proporciona una renderización eficiente del

2.2 Arquitectura del Sistema (MVC)

El proyecto se estructura bajo el patrón Modelo-Vista-Controlador (MVC),
garantizando una separación clara de responsabilidades:

    •   Modelo: Define la estructura de los datos (platos, reservas, stock), utilizando el
        ORM Eloquent para interactuar con MySQL mediante consultas seguras.

    •   Controlador: Actúa como el motor de la aplicación, gestionando la lógica de
        negocio, como la validación de stock (available_servings) y la comunicación con
        Apis externas.

    •   Vista: Presenta la información procesada al usuario, con lógica integrada para
        adaptar la interfaz según el estado del servicio (ej. mostrar "plato agotado").

2.3 Seguridad y Protección de Datos

La seguridad es un pilar fundamental en el diseño de El Fogón Dominicano:

    •   Protección contra Inyecciones SQL: Gracias al uso de Eloquent y PDO
        parameter binding, todas las consultas a la base de datos están protegidas
        contra inyecciones, neutralizando cualquier entrada malintencionada por parte
        del usuario.



    •   Gestión de Concurrencia (Integridad de Inventario): Para prevenir la
        sobreventa, se han implementado transacciones de base de datos
        (DB::transaction). Esto garantiza que la actualización del stock y la creación
        de la reserva sean procesos atómicos, evitando errores de consistencia cuando
        múltiples usuarios reservan simultáneamente.



    •   Gestión del Consentimiento (RGPD): Bajo el principio de Privacy by Design, el
        formulario de reserva incluye una validación de consentimiento obligatorio. Si el

        técnicamente la acción, asegurando el cumplimiento normativo.

2.4 Estandarización de Infraestructura

Para asegurar que el proyecto funcione de manera idéntica desde el desarrollo hasta la
producción:

    •   Contenedorización (Docker): El entorno completo (PHP, MySQL, servidor web)
        está empaquetado en contenedores aislados. Esto garantiza portabilidad total y
        elimina los fallos de configuración entre entornos.



    •   Despliegue profesional (Cloud + dominio propio): El proyecto está preparado
        para desplegarse en un proveedor cloud (por              ejemplo, AWS o DigitalOcean) y
        asociarse a un dominio propio. La configuración de entorno (APP_URL, base de
        datos, correo, etc.) se gestiona mediante variables .en, lo que permite mover la
        aplicación entre entornos sin cambios de código.

3.2 Optimización de la experiencia de usuario (UX) ante el agotamiento de stock

El sistema debía gestionar la frustración del usuario cuando el menú diario se agota. La
problemática residía en cómo comunicar esta información de manera elegante y
funcional sin romper el flujo de navegación.

    •    Resolución: Mediante la lógica en las vistas Blade, se automatizó una transición de
         estado: cuando la variable available_servings llega a cero, el sistema oculta
         automáticamente el formulario de reserva y muestra una interfaz de "plato agotado".
         Esto permite al usuario conocer el estado del servicio en tiempo real, evitando intentos
         fallidos de reserva y mejorando la transparencia del servicio.

























3.3 Estabilización del rendimiento y gestión de tiempos de espera

Durante el desarrollo, se presentaron desafíos relacionados con los tiempos de
respuesta del servidor (errores de tipo 504), causados por una gestión ineficiente de la
carga de recursos o procesos bloqueantes.

    •    Resolución: Se procedió a una depuración integral del flujo de carga de la página. Esto
         incluyó la limpieza de clases CSS obsoletas que generaban conflictos de renderizado,
         la optimización de las consultas a base de datos y la implementación de una jerarquía
         de estilos más ligera y sólida. Como solución definitiva al estilo visual, se aplicaron
         directivas de alta prioridad (!important) en las hojas de estilo mediante Tailwind CSS

3.4 Adaptabilidad y personalización de interfaces

El sistema presentaba una dificultad para heredar estilos predefinidos en
componentes de la plantilla, lo que impedía una personalización visual completa.

    •   Resolución: Se adoptó un enfoque de personalización agresiva mediante el uso de
        utilidades de estilo directo, permitiendo una identidad visual coherente con la
        propuesta de El Fogón Dominicano. Este paso fue fundamental para asegurar que,
        independientemente de las actualizaciones de la plantilla base, el diseño se
        mantuviera fiel a la marca.

3.5 Protección contra Inyecciones SQL

El sistema utiliza Eloquent (Laravel), que ejecuta consultas parametrizadas (prepared
statements) mediante PDO. Esto reduce de forma significativa el riesgo de inyección SQL, ya
que los valores del usuario no se concatenan directamente en las consultas. Como buena
práctica, se limita el uso de consultas "raw" y, cuando son necesarias, se emplean siempre
parámetros enlazados y validación de entrada.

3.6 Gestión de Concurrencia y Prevención de "Sobre-reserva"

Para evitar la sobreventa de raciones (cuando varios usuarios intentan reservar la
última ración disponible simultáneamente), se ha implementado una lógica de control
estricta:

    •   Validación en Servidor: Antes de persistir cualquier reserva, el sistema ejecuta
        una validación de available_servings dentro de una transacción de base
        de datos (DB::transaction).

    •   Bloqueo de Registro (Atomicidad): La operación de reserva y la actualización
        del stock se ejecutan como un proceso atómico. Si dos usuarios intentan
        reservar al mismo tiempo, la base de datos garantiza que solo la primera
        transacción sea exitosa; la segunda es rechazada automáticamente al detectar

        inventario.

Apartado 3.9: Integridad Reversible del Stock (Gestión de
Cancelaciones)

Un sistema de reservas robusto no solo debe preocuparse por el descuento de stock,
sino también por su recuperación ante cancelaciones o eliminaciones de pedidos. Sin
esta lógica, el sistema sufriría de "stock fantasma", mostrando platos agotados
cuando, en realidad, las raciones han vuelto a estar disponibles tras una baja.

    •   El Reto Técnico: Garantizar que la eliminación de un registro en la tabla de
        reservas dispare automáticamente una actualización en la tabla de platos. El
        principal riesgo es la inconsistencia: que el pedido se           borre, pero el stock no se
        actualice debido a un error en medio del proceso.

    •   La Solución (Lógica de Desacoplamiento):               Se implementó una lógica de
        "limpieza" en el controlador de administración. Antes de proceder a la
        eliminación física o lógica del pedido, el sistema recupera la cantidad de
        raciones asociadas y ejecuta un método de incremento (               increment()) sobre el
        plato correspondiente.

    •   Implementación de Transacciones:               Al igual que en la creación de reservas,
        este proceso se realiza bajo una        transacción de base de datos          . Si el

                                                                                                        13


===PAGE 14===
        incremento del stock falla por cualquier motivo técnico, la eliminación del
        pedido se revierte automáticamente (rollback), asegurando que nunca se pierda
        el rastro de una venta sin que su stock esté debidamente cuadrado.

    •   Impacto en el negocio: Esta lógica mantiene la consistencia entre pedidos y
        raciones disponibles, evitando "stock fantasma" y mejorando la fiabilidad del
        inventario mostrado al cliente.



Apartado 4: Infraestructura y Entorno de Ejecución

4.1 Estandarización del entorno con Docker

Para evitar las discrepancias entre los entornos de desarrollo (local) y producción, el
proyecto ha sido contenedorizado utilizando Docker. Esta decisión permite encapsular
el servidor web, el intérprete de PHP, la base de datos y todas las dependencias
necesarias en un ecosistema controlado y aislado del sistema operativo anfitrión.

4.2 Gestión mediante Docker Compose

El despliegue y la orquestación del proyecto se gestionan mediante docker-
compose.yml, lo cual proporciona los siguientes beneficios estratégicos:

    •   Consistencia absoluta: El entorno de ejecución es idéntico en cualquier
        máquina donde se despliegue el proyecto, eliminando el fallo recurrente de "en
        mi máquina funciona".

    •   Gestión simplificada de servicios: Permite levantar la infraestructura
        completa con un único comando (docker-compose up), facilitando tanto la
        tarea de desarrollo como la de puesta en marcha en servidores remotos.

    •   Portabilidad: La configuración definida permite que el proyecto sea
        desplegable en cualquier infraestructura moderna que               capaz de soportar












                                                                                                       14


===PAGE 15===
        contenedores, otorgando una gran libertad de elección de proveedores de
        hosting (como AWS, DigitalOcean o proveedores locales).






















4.3 Integridad del Ciclo de Vida del Software

La implementación de Docker mejora la estabilidad del sistema ante cambios de
máquina o de servidor. Al separar los servicios en contenedores (aplicación y base de
datos; y, cuando aplica, servicios auxiliares), se reduce el acoplamiento con el sistema
operativo anfitrión y se facilita el mantenimiento, el despliegue y la recuperación ante
incidencias.

eficazmente a una necesidad demográfica real en el Vallès Oriental, eliminando las barreras de
comunicación y garantizando una experiencia de usuario profesional.

6.3 Lecciones Aprendidas

El proceso de desarrollo ha proporcionado un aprendizaje integral, reforzando competencias

6.4 Visión de futuro

El Fogón Dominicano se plantea como un MVP (Minimum Viable Product) con una hoja de ruta
clara para su evolución hacia una plataforma 4.0:

    •    Evolución Comercial y Pasarela de Pagos: Integración de APIs de pago (Stripe/PayPal)
         para permitir el prepago de pedidos, reduciendo la tasa de cancelaciones y
         optimizando el flujo de caja.

    •    Logística Avanzada y Geolocalización:
    •     Implementación: Uso de la Geolocation API del navegador para capturar la ubicación
         exacta del cliente en pedidos a domicilio.

    •    Impacto: Permitirá optimizar las rutas de reparto y calcular tiempos de entrega
         precisos basados en el tráfico real, mejorando la eficiencia operativa y la satisfacción
         del usuario.

    •    Monitorización en Tiempo Real (WebSockets):
         Implementación: Integración de Laravel Reverb o Pusher para establecer conexiones
         bidireccionales.
    •    Impacto: El cliente podrá visualizar el estado de su pedido (En preparación, En reparto,
         Entregado) en tiempo real sin necesidad de refrescar la página. Esto elimina la
         ansiedad del usuario y profesionaliza el servicio de atención al cliente.
